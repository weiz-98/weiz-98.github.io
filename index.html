  • const catSchema = new Schema(...): 定義了 Cat 的 Mongoose 模型結構，設定了 name 和 weight 兩個字段都為必須的，並啟用了自動時間戳記錄（timestamps: true），這會自動添加 createdAt 和 updatedAt 這兩個字段。
  • export default mongoose.models.Cat || model('Cat', catSchema): 導出 Cat 模型。這條語句首先檢查 mongoose.models 中是否已經存在 Cat 模型（避免重複定義），如果不存在，則使用 model('Cat', catSchema) 創建新的模型。
  • 在 mongoose.models.Cat 中，Cat 指的是 Mongoose 的模型。當你在 Mongoose 中定義一個模型時，例如通過 mongoose.model('Cat', catSchema)，Mongoose 會在其內部 models 對象上註冊該模型。所以 mongoose.models.Cat 是在查找由 mongoose.model 創建並註冊到 Mongoose 的模型。
  • 在 import { Cat } from '../types/cat' 中，Cat 指的是一個 TypeScript 類型，它在 ../types/cat 文件中定義。這個類型用於在 TypeScript 中進行靜態類型檢查，並不會在運行時存在。

這兩者是相關聯的，但用於不同的目的：

  • Mongoose 的 Cat 模型用於與 MongoDB 交互，並且會包含像是查詢、保存、更新文檔等方法。
  • TypeScript 的 Cat 類型用於在編譯時期檢查代碼，確保你正確地使用與貓咪相關的資料結構。

repo/cat.ts

  • 定義資料庫操作函數：
    • getCats: 定義一個函數，當被調用時，它會從資料庫中找到所有的貓咪資料並返回一個承諾（Promise）。
    • addCat: 接受一個貓咪物件作為參數，該物件是 Cat 型別的部分集合（使用了 Partial），並創建一個新的貓咪記錄在資料庫中。
    • getCatById: 根據給定的 ObjectId 從資料庫中尋找一個貓咪的資料。
    • updateCatById: 接受一個 ID 和一個更新物件，該物件也是 Cat 型別的部分集合，並更新相應的貓咪記錄。
    • deleteCatById: 根據提供的 ID 從資料庫中刪除一個貓咪的記錄。
  • await CatModel.findOneAndUpdate({ _id: id }, update, { new: true }): 異步等待 Mongoose 的 findOneAndUpdate 方法的執行。這個方法尋找一個匹配 _id 的文檔，用 update 物件進行更新。
  • new: true 是一個選項，當設置為 true 時，findOneAndUpdate 方法將會返回更新後的文檔。如果不設置或者設為 false，它會返回原始文檔。
  • 當 findOneAndUpdate 方法在資料庫中匹配不到任何文檔時，它將返回 null。由於這個方法本身返回一個 Promise，因此當沒有找到匹配的文檔時，你實際上得到的是一個解析為 null 的 Promise。
  • Types.ObjectId 是 Mongoose 中的一種數據類型，對應於 MongoDB 中的 ObjectId。每個在 MongoDB 中儲存的文件都有一個唯一的 _id 屬性，其值為 ObjectId。這個值是自動生成的，並且通常用於檢索和操作特定的文檔。

server.ts

  • 路由處理：
    • server.get('/ping', ...): 定義一個 GET 請求路由，當訪問 /ping 時，將會重定向到 /other 路由。
    • server.get('/other', ...): 定義一個 GET 請求路由，返回一個包含消息的對象。
    • server.get('/cats', ...): 定義一個 GET 請求路由，從資料庫中獲取所有貓咪數據。
      • 如果成功獲取到貓咪資料，則通過 reply.status(200).send({ cats }) 將 HTTP 狀態碼設置為 200（OK），並將貓咪數據以 JSON 格式發送給客戶端。
      • getCats() 函數返回的結果被存儲在常量 cats 中，這個常量被類型註釋為 Array<Cat>，這意味著它是一個包含 Cat 型別物件的數組。
    • server.post('/cats', ...): 定義一個 POST 請求路由，允許客戶端添加一個新的貓咪。
      • const catBody = request.body as Cat：從請求的 body 中提取貓咪資料，並假定它符合 Cat 型別。使用 as 關鍵字是 TypeScript 的型別斷言，它告訴編譯器我們相信 request.body 的結構與 Cat 型別匹配。
      • const cat: Cat = await addCat(catBody)：調用 addCat 函數，將 catBody 傳入，以在資料庫中創建新的貓咪紀錄。這個函數是異步的，因此使用 await 來等待其完成。
      • return reply.status(200).send({ cat })：如果貓咪成功創建，將 HTTP 狀態碼設置為 200，並將新創建的貓咪資料發送給客戶端。
    • server.get('/cats/:_id', ...): 定義一個 GET 請求路由，用於根據 ID 獲取特定的貓咪。
      • const { _id } = request.params as CatParams：從請求參數中提取 _id。CatParams 接口確保 _id 是 Types.ObjectId 的實例。
      • const cat = await getCatById(_id)：使用提取的 _id 調用 getCatById 函數，從資料庫中獲取對應的貓咪資料。
      • return reply.status(200).send({ cat })：如果找到貓咪資料，則以 200 狀態碼回應，並附上貓咪資料。
    • server.patch('/cats/:_id', ...): 定義一個 PATCH 請求路由，用於更新特定 ID 的貓咪資料。
      • server.patch('/cats/:_id', async (request, reply) => {...})：定義了一個處理 PATCH 請求的路由。PATCH 請求通常用於執行部分更新操作。
      • const { _id } = request.params as CatParams：從請求 URL 中提取貓咪的 _id。
      • const update = request.body as Partial<Cat>：從請求的 body 中提取更新的資料，使用 Partial<Cat> 表示這些資料可能只包含 Cat 型別的一部分。
      • const cat = await updateCatById(_id, update)：調用 updateCatById 函數，傳入 _id 和更新的資料，以更新資料庫中的貓咪紀錄。
      • return reply.status(200).send({ cat })：如果更新成功，回應 200 狀態碼和更新後的貓咪資料。
    • server.delete('/cats/:_id', ...): 定義一個 DELETE 請求路由，用於刪除特定 ID 的貓咪。
  • serverOf: 這是一個工廠函數，用於創建 Fastify 的實例。
  • 在 serverOf 函數中定義 CatParams 接口的目的是為了給 request.params 提供一個明確的型別。這樣做可以讓 TypeScript 編譯器知道，當從 request.params 中提取值時，這些值應該遵循 CatParams 接口的結構。這提高了代碼的類型安全性，並有助於在編譯時捕捉錯誤，例如當你嘗試訪問 CatParams 未定義的屬性時。
  • request 和 reply 是 Fastify 提供的兩個對象：
    • request: 代表著入站的請求，包含了路徑、查詢參數、標頭、body 等。
    • reply: 提供了多種方法來構造和發送 HTTP 回應。

使用 request.params as CatParams 的原因：

  • 在 server.get('/cats/:_id', ...) 路由中，我們通過 URL 的參數 _id 獲取特定的貓咪數據。URL 參數是通過路由模板 /cats/:_id 定義的，其中 ==:_id 是一個動態參數==。
  • ==request.params 包含了所有的路由參數作為其屬性==，這些參數是在 URL 路徑中解析出來的。
  • as CatParams 是一個 TypeScript 的類型斷言，它告訴 TypeScript 編譯器 request.params 對象匹配 CatParams 這個接口。
  • CatParams 通常用於路由參數的類型，可能包含多個值，比如在一些複雜的路由中。

使用 request.body as Cat 的原因：

  • 在 server.post('/cats', ...) 路由中，我們從客戶端接收到的貓咪數據是通過 HTTP 請求的 body 傳遞的，這通常是在執行 POST 或 PUT 請求時發生的。
  • ==request.body 包含了 POST 請求的數據主體==，這些數據是客戶端發送的。
  • as Cat 告訴 TypeScript request.body 對象應該符合 Cat 類型，這是在 types/cat.ts 中定義的。

為什麼可以直接使用 { _id }：

  • 在 const { _id } = request.params as CatParams 中，這是 JavaScript 的解構賦值語法。這意味著從 request.params 對象中提取 _id 屬性並創建一個同名的 _id 變量。
  • 解構賦值允許你從一個物件中提取出一個或多個屬性，並將它們賦值給新的變量，變量的名稱直接使用屬性的名稱。
  • 在這種情況下，不需要額外定義一個變量名稱，因為屬性名 _id 和變量名 _id 是一致的，這樣可以簡化代碼。

補充

當前端（客戶端）發送一個 POST 請求到 /cats 端點時，request.body 將包含前端發送的數據。在你的 Fastify 服務器中，這個數據被期望符合 Cat 類型的結構，這通常是由前端按照約定的格式發送的 JSON 對象。

在處理 request.body 時，使用 as Cat 是在告訴 TypeScript：“我知道這個物件的結構符合 Cat 類型”。這是一種類型斷言，它不會改變 request.body 的值，也不會在運行時進行任何類型檢查。這只是在編譯階段提供給 TypeScript 的一個提示，以便於在開發中避免類型相關的錯誤。

如果你不寫 as Cat，那麼 TypeScript 將不會對 request.body 進行特定的類型檢查，這可能會在你的代碼中隱藏潛在的錯誤，因為 request.body 可能包含任何結構的數據。

為了更安全地處理這類情況，你可以做以下的修改：

  1. 進行運行時檢查：在將 request.body 斷言為 Cat 之前，先檢查它是否真的符合 Cat 的結構。這通常涉及到檢查 request.body 是否有 name 和 age 屬性，並且它們的類型是否正確。
  1. 使用驗證庫：使用像 Joi 或 Yup 這樣的數據驗證庫來驗證和解析 request.body，這樣你就可以更有信心地知道數據的結構是正確的。

這些方法會增加代碼的健壯性，並減少因為錯誤的數據結構而導致的錯誤。在生產環境的應用程序中，這是非常推薦的做法。
